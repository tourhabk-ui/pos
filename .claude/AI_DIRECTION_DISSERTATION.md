## ГЛАВА 1. ПОСТАНОВКА ПРОБЛЕМЫ

### Почему туристический бизнес на Камчатке особенный

Камчатка — это не Мальдивы. Это экстремальная логистика:
- 200+ операторов, 1189+ маршрутов, разрозненных в пространстве и времени
- Погода меняет планы за 40 минут
- МЧС закрывает маршруты без предупреждения
- Туристы из 40+ стран говорят на разных языках
- Гиды появляются и исчезают как сезонные явления
- Сплавы зависят от уровня воды в реке, не только от расписания

**Один человек-менеджер не может:**
- Одновременно отслеживать погоду на 1189 маршрутах
- Знать кто сейчас свободен из 200+ гидов
- Помнить историю 5000 туристов
- Реагировать на МЧС-оповещения в 3 часа ночи
- Одновременно вести переговоры с 30 операторами

**Вывод:** Платформа без AI обречена быть справочником. С AI — становится живым организмом.

## ГЛАВА 2. КОНЦЕПЦИЯ AI ДИРЕКЦИИ

### Что это такое

AI Дирекция — это **коллектив специализированных AI-агентов**, каждый из которых является экспертом в своей области и действует автономно в рамках своих полномочий.

Аналогия: реальная дирекция компании.

```
CEO (директор)           → PlatformAgent    — единая точка принятия решений
Директор по продажам     → LeadAgency        — лиды, конверсия, CRM
Директор по операциям    → OperatorAgency    — туры, билеты, слоты
Директор по безопасности → RescueAgency      — МЧС, риски, SOS
Директор по маркетингу   → MarketingAgency   — контент, SEO, Telegram
Юрист                    → LegalAgency       — договоры, согласования
Аудитор                  → QualityAgency     → отзывы, NPS, жалобы
HR/обучение              → EvolutionAgency   — обучение других агентов
Служба безопасности      → SecurityAgency    — мошенники, атаки
Эколог                   → EcoAgency         — ООПТ, экологические нормы
```

Каждый агент:
- **Видит только своё** — имеет доступ только к релевантным данным
- **Действует в своих полномочиях** — есть что-то что он может делать сам, есть ApprovalRequired
- **Учится** — фиксирует успешные/неуспешные паттерны
- **Общается с другими** — через AgentMesh

## ГЛАВА 3. АРХИТЕКТУРА СИСТЕМЫ

### Поток обработки запроса

```
Пользователь: "Покажи туры с местами на 15 июня для группы 6 человек, рыбалка"
                              ↓
                    PlatformAgent.dispatch()
                              ↓
              IntentClassifier (keyword → AI fallback)
                         ↓            ↓
               "tourist_recommend"  "op_tours_summary"
                              ↓
                    ContextHub.build()
              [userId, role, history, sessionId]
                              ↓
                 Загрузка AgentMemory (что знаем об этом юзере?)
                              ↓
                ExperimentTracker (A/B тест активен?)
                              ↓
                    Agency.handle(context)
                              ↓
                ObservationLogger.log() + ExperimentTracker.record()
                              ↓
                    Ответ пользователю
```

### 40+ интентов — что они значат

**Операторские (op_*):**
```
op_tours_summary     — "покажи мои туры" → SQL: SELECT * FROM operator_tours WHERE operator_id=$1
op_bookings_today    — "что за брони сегодня" → SELECT * FROM operator_bookings WHERE date=TODAY
op_revenue           — "выручка за 7 дней" → SUM(final_price) GROUP BY date
op_create_tour       — "создай тур Рыбалка" → INSERT INTO operator_tours (ApprovalRequired)
op_fill_ai           — "заполни тур 3" → POST /api/operator/tours/auto-fill-ai
op_add_slots         — "добавь слоты на июль" → INSERT INTO tour_availability
```

**Туристические:**
```
tourist_recommend    — "что посмотреть на Камчатке вулканы" → recommendations engine
```

**Спасательные:**
```
rescue_sos_stats     — "сколько SOS за месяц?" → COUNT FROM sos_requests
rescue_weather_risk  — "опасен ли маршрут на Авачу завтра?" → safety API + weather
rescue_protocols     — "что делать при встрече с медведем?" → knowledge base
```

**Административные:**
```
admin_digest         — утренний дайджест Platform → leads/bookings/alerts сводка
admin_leads_summary  — анализ лидов за период
admin_ops_report     — операционный отчёт
```

## ГЛАВА 4. ПРАКТИЧЕСКОЕ ПРИМЕНЕНИЕ

### 4.1. Оператор: Ежедневная работа без Excel

Сейчас оператор **"Камчатская рыбалка"**:
- Утром открывает Telegram — 5 сообщений с вопросами
- Открывает блокнот — сверяет даты
- Пишет ответы вручную по 20 минут каждый

С AI Дирекцией (оператор пишет в чате):

```
Оператор: "Что у меня сегодня?"
AI: "Сегодня 3 брони: Иванов (4 чел, 09:00), Петров (2 чел, 14:00),
     Группа из Москвы (8 чел, 09:00 — конфликт с Ивановым!).
     Рекомендую: перенести Иванова на 13:00."

Оператор: "Создай тур Медведи на Курильском июль-август"
AI: [ApprovalRequired] "Подготовил черновик:
     Название: Наблюдение за медведями, Курильское озеро
     Цена: 45 000 ₽ (рыночная — 40-55k)
     Вместимость: 8 чел
     Длительность: 2 дня
     Подтвердить?"

Оператор: "Заполни описание тура 5"
AI: [через GPT-4o] "Готово: написал описание 800 символов,
     добавил 3 фото-теги, SEO-заголовок, список что включено."
```

## ГЛАВА 5. ЧЕГО НЕ ХВАТАЕТ (ЧЕСТНЫЙ АНАЛИЗ)

### Работает прямо сейчас
```
✓ PlatformAgent.dispatch() — полный поток
✓ IntentClassifier — keyword matching для 40+ интентов
✓ AdminAgency — digest, leads_summary
✓ OperatorAgency — tours_summary, bookings, revenue, fill_ai, add_slots
✓ TouristAgency — recommend (через trip-recommender)
✓ RescueAgency — protocols, weather_risk
✓ ExperimentTracker — A/B тесты на интенты
✓ AgentMemory — персистентная память агентов
✓ ObservationLogger — логирование всех действий
✓ Permissions — role-based intent gating
```

### Не работает / сломано
```
✗ op_fill_ai: требует NEXT_PUBLIC_SITE_URL на проде — не проверено
✗ rescue_sos_stats: запрашивает таблицу sos_requests — существует?
✗ MarketingAgency: заглушки, нет реального постинга в Telegram
✗ LegalAgency: базовые ответы, без конкретики по Камчатке
✗ EcoAgency: нет данных об ООПТ в БД
✗ AgentMesh: архитектура есть, реальные межагентные вызовы не реализованы
✗ Endpoint /api/agents/dispatch: только для admin — нет tourist endpoint
```

## ГЛАВА 6. ПЛАН РАЗВИТИЯ

### Версия 1.0 (сейчас, уже задеплоена)
```
Один оператор → один агент → один ответ
Оператор спрашивает → OperatorAgency отвечает
```

### Версия 2.0 (2-4 недели)
```
Многошаговые диалоги: агент помнит контекст 10 сообщений
Межагентные вызовы: OperatorAgency → RescueAgency (проверить погоду перед созданием тура)
Проактивные уведомления: агент сам пишет оператору при изменении ситуации
Tourist endpoint: туристы получают AI-ассистента напрямую
```

### Версия 3.0 (1-3 месяца)
```
Полная автономия: агент анализирует рынок, предлагает изменение цен
"Цена на Авачинский у конкурентов выросла на 15% — рекомендую поднять вашу с 8 500 до 9 500₽"
Предиктивный спрос: "Через 2 недели праздники — заполните слоты на 8-10 мая"
Качество: AI обнаружил что у тура 3 низкий рейтинг из-за плохих фото — предлагает обновить
```

### Версия 4.0 (3-6 месяцев)
```
Полная автономия управления операционным слоем
Агент — полноценный директор по продажам
Работает 24/7 без участия человека на рутинных задачах
Человек только одобряет нестандартные решения (ApprovalRequired)
```

## ГЛАВА 7. ПОЧЕМУ ЭТО ВАЖНО

### Камчатка vs остальной мир туризма

| Аспект | Традиционный туризм | Камчатка |
|--------|---------------------|---------|
| Стабильность | Высокая | Низкая (погода, вулканы) |
| Предсказуемость | Да | Нет |
| Автоматизация | Возможна шаблонно | Требует интеллекта |
| Конкуренты | Booking.com, Airbnb | Нет аналогов |

**Уникальная позиция:** в мире нет AI-powered платформы для экстремального туризма.
Booking.com не умеет работать с МЧС-алертами. Viator не знает что такое лавинная угроза.

Мы строим не просто агрегатор. Мы строим **интеллектуальную операционную систему для дикой природы**.

### Что мы докажем рынку
1. AI может управлять туристическим бизнесом в условиях непредсказуемости
2. Маленький оператор с AI-дирекцией конкурирует с крупной компанией
3. Безопасность туристов повышается когда система знает всё в реальном времени
